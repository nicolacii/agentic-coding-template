# Skill: Receiving Code Review

> Реакция на findings ревью — техническая оценка, не перформативное согласие.
> Источник: Superpowers `receiving-code-review`. Запуск: `/receiving-review` или после `/review`.

---

## Когда вызывать

- Получил findings от `/review` / whole-branch ревьюера / внешнего ревью / PR-комментов.
- ПЕРЕД тем как менять код по фидбеку.

---

## Паттерн реакции (по порядку)

```
1. READ       — прочитать весь фидбек целиком, не реагируя по пунктам.
2. UNDERSTAND — переформулировать требование своими словами (или спросить).
3. VERIFY     — проверить против реальности кодовой базы.
4. EVALUATE   — технически верно для ЭТОЙ кодовой базы/стека?
5. RESPOND    — техническое подтверждение ИЛИ обоснованный push back.
6. IMPLEMENT  — по одному пункту, тестировать каждый.
```

**Непонятный пункт → СТОП, не реализовывать ничего, спросить.** Пункты связаны;
частичное понимание = неверная реализация.

## Проверка перед реализацией (особенно внешний фидбек)

- [ ] Технически верно для этого стека/версии/платформы?
- [ ] Не ломает существующую функциональность?
- [ ] Есть ли причина текущей реализации (legacy/совместимость)?
- [ ] Понимает ли ревьюер полный контекст?

**YAGNI-чек:** ревьюер просит «сделать properly»? → `grep` использования.
Не используется → «этот код не вызывается, удалить (YAGNI)?». Используется → делать properly.

## Когда push back (с техническим обоснованием, не защищаясь)

- Предложение ломает существующее.
- Ревьюер без полного контекста.
- Нарушает YAGNI (фича не нужна).
- Технически неверно для стека.
- Конфликтует с ранее принятым решением пользователя → СТОП, обсудить с пользователем.

Как: техническое обоснование + конкретные вопросы + ссылка на работающие тесты/код.
Был неправ в push back → «проверил X, действительно Y, делаю».

## Порядок исправлений

1. Сначала уточнить непонятное.
2. Затем по приоритету: **blocking (ломает/security) → simple (typo/import) → complex (рефактор/логика)**.
3. **Тестировать каждый фикс отдельно** (см. `/verify` Gate Function), проверить отсутствие регрессий.

## Признать верный фидбек — действиями

✅ «Исправил. [что изменил]» / «Поймал — [issue]. Фикс в [location]».
❌ Перформативное согласие («ты абсолютно прав», «отличный поинт», благодарности) — опционально к стилю Nik, не обязательно. Главное: просто исправь и покажи в коде.
