# Skill: Verify — Полная верификация перед "готово"

> Комбинация Cross-check + Challenge + Confidence + Forbidden. Вызывается ПЕРЕД каждым DONE блоком.
> Запуск: `/verify` или автоматически при ШАГ 3 (Проверка) из core-rules.md
> **ОБЯЗАТЕЛЕН для КАЖДОГО ответа с DONE блоком.**

---

## Когда вызывать

- Перед КАЖДЫМ "готово" / DONE блоком
- После завершения любой задачи
- При сомнениях в качестве результата

---

## 1. CROSS-CHECK — Независимая проверка

**НЕ говори "готово" без независимой проверки!**

### Обязательно:
- Каждый созданный/изменённый файл **ОТКРЫТЬ и ПРОЧИТАТЬ**
- Проверить результат **ДРУГИМ методом** (не тем которым делал)
- Заполнить таблицу артефактов
- Сравнить результат с **буквальным** запросом пользователя

### Методы Cross-check:

| Сделал | Проверка |
|--------|----------|
| Создал файл | Открой и прочитай |
| Изменил код | Запусти linter |
| API endpoint | curl + прочитай код |
| Данные | pandas + SQL |
| UI | Визуальная проверка |

### Формат:

```markdown
| Файл | Открыт | Проверен |
|------|--------|----------|
| [путь] | ✅/❌ | ✅/❌ |
```

**Если Cross-check не выполнен → Уверенность -30%.**
**Два метода — если проверял одним, проверь другим.**
**Gaps — если нашёл расхождение, ИСПРАВЬ перед "готово".**

---

## 2. CHALLENGE — 4 вопроса (ОБЯЗАТЕЛЬНО)

```
1. Как опровергнуть мой вывод?     → [ответ]
2. Все файлы открыл и проверил?    → [ДА/НЕТ]
3. Какие edge cases пропустил?     → [список]
4. Job пользователя решён?         → [ДА/НЕТ]
```

### Decision Gates (жёсткие правила):

- **Вопрос 2 = НЕТ** → СТОП. Открой файлы. Потом "готово".
- **Вопрос 4 = НЕТ** → СТОП. Доделай. Потом "готово".
- **Не можешь ответить** → НЕ ГОТОВО.
- **НЕ говорить** "Готово" без ответов на 4 вопроса.
- **НЕ говорить** "Успешно" без проверки.
- **НЕ говорить** "Работает" без тестирования.

---

## 3. CONFIDENCE — Калибровка уверенности

### Формула:

```
100% минус:
• Код не тестировал:     -50%
• Файлы не открывал:     -40%
• Cross-check не делал:  -30%
• Предположения:         -25%
```

### Пороги:

| Уверенность | Действие |
|-------------|----------|
| ≥ 90% | Можно завершать |
| 70-89% | Завершать с оговорками |
| 50-69% | Нужна дополнительная проверка |
| < 50% | СТОП — спросить пользователя |

### Формат:
```
**Уверенность: [X]% — [причина]**
```

**CRITICAL:**
- НЕ завышать уверенность — честно применять вычеты
- НЕ объявлять "готово" при уверенности < 80%
- ПОВЫСИТЬ уверенность через: тесты, cross-check, открытие файлов
- Пересчитывать после КАЖДОГО значимого действия

### Примеры:

GOOD: Честная калибровка
```
Уверенность: 85% → 95% — после cross-check и запуска тестов
```

GOOD: Снижение при отсутствии проверки
```
Уверенность: 95% → 45% — код написан, но не запускал (тесты -50%)
```

BAD: Завышенная
```
Уверенность: 95% — всё должно работать
```
Реальность: файлы не открывались (-40%), код не тестировался (-50%). Реальная уверенность: 10%.

## 4. FORBIDDEN — Критические запреты

**При нарушении: СТОП → Исправить → Продолжить.**

- ❌ НИКОГДА файлы с `_fixed`, `_final`, `_v2` — редактируй исходник
- ❌ НИКОГДА перезаписывать файл целиком — точечные изменения
- ❌ НИКОГДА создавать дубли — найди существующий
- ❌ НИКОГДА "Готово" без открытия файлов
- ❌ НИКОГДА "Работает" без запуска
- ❌ НИКОГДА игнорировать ошибки терминала
- ❌ НИКОГДА предположения без проверки в данных

### Перед "готово" проверь:
- [ ] Все файлы открыты и проверены?
- [ ] Ошибок в терминале нет?
- [ ] Cross-check выполнен?

**Если хоть один ❌ → не говори "готово"**

---

## 5. RAT — Riskiest Assumption Test (для STANDARD/COMPLEX)

### 3 шага:
1. **ВЫПИСАТЬ** все предположения
2. **ОТРАНЖИРОВАТЬ** по риску (что "убьёт" решение?)
3. **ПРОВЕРИТЬ** TOP риск ПЕРЕД кодингом

### Категория 0 (ВСЕГДА ПЕРВЫЙ): "Я правильно понял запрос?"

Проверка:
1. Перечитать запрос **БУКВАЛЬНО**
2. Выписать explicit constraints **дословно**
3. Проверить: я "интерпретирую" или "следую"?

Red Flags:
- "Пользователь имел в виду..." — интерпретация
- "Логично предположить..." — додумывание
- "Для удобства я..." — самодеятельность

### Категории рисков:
| Категория | Примеры |
|-----------|---------|
| Техническая | Подход подходит? Не сломает существующее? |
| Данные/API | Структура как ожидается? Типы совместимы? |
| UI/UX | Компонент существует? Responsive нужен? |
| Зависимости | Версии совместимы? Entry point полный? |

**Если TOP-1 риск опровергнут → ПЕРЕСМОТРЕТЬ ПЛАН.**
