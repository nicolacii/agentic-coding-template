# Skill: Writing Skills (и правил)

> **Skills — это TDD, применённый к процессной документации.** Ты пишешь тест (pressure-сценарий
> с субагентом), смотришь как он падает (baseline-поведение без скилла), пишешь скилл, смотришь как
> проходит (агент соблюдает), рефакторишь (закрываешь лазейки).
> Источник: Superpowers `writing-skills`. Вызывается при создании/правке скилла или из reflection→rules (`/backlog-to-rules`, `/reflection`).

**Core principle:** Если ты не видел, как агент проваливается БЕЗ скилла, ты не знаешь, тот ли урок скилл преподаёт.

---

## Когда создавать скилл

**Создавать когда:**
- Приём не был очевиден сходу
- Будешь ссылаться на это в разных проектах
- Паттерн применим широко (не проектно-специфичный)

**НЕ создавать для:**
- Одноразовых решений
- Стандартных, хорошо документированных практик
- Проектных конвенций → в `<project>/CLAUDE.md`, не в глобальный скилл
- Механических ограничений, которые лучше форсить регэкспом/валидацией (автоматизируй, не документируй)

## ⭐ Match the Form to the Failure (ключевой инсайт)

Перед тем как писать guidance — **классифицируй baseline-провал**. Форма, которая бронирует один тип
провала, измеримо ухудшает другой. Не тянись к запрету по умолчанию.

| Baseline-провал | Правильная форма | Неправильная форма |
|---|---|---|
| Нарушает правило под давлением (знает, но делает) | **Prohibition + rationalization table + red flags** (см. Bulletproofing) | Мягкое «предпочтительно…», «рассмотри…» |
| Соблюдает, но выход не той формы (раздутый промпт, спрятанный вердикт, пересказ спеки) | **Positive recipe / contract:** укажи, чем выход ЯВЛЯЕТСЯ — его части, по порядку | Список запретов («не пересказывай», «никогда не…») |
| Пропускает обязательный элемент того, что и так производит | **Structural:** REQUIRED-поле/слот в шаблоне, который он заполняет | Prose-напоминания рядом с шаблоном |
| Поведение должно зависеть от условия | **Conditional** на наблюдаемом предикате («если brief существует — ссылайся на него») | Безусловное правило + пункты-исключения |

**Почему запреты ломают shaping-проблемы:** под конкурирующим стимулом («сделай промпт
самодостаточным») агенты торгуются с «не делай X». В head-to-head тестах prohibition-версия давала
БОЛЬШЕ нежелательного контента, чем recipe-версия, и хуже, чем вообще без guidance. Recipe не
оставляет предмета для торга: выход либо соответствует форме, либо нет.

**Правила для любой выбранной формы:**
- **Никаких nuance-clauses.** «Не делай X, если это не важно» переоткрывает торг. Реальное
  исключение — отдельный conditional на наблюдаемом предикате.
- **Exemption-clauses не скоупятся.** «Это ограничение не касается блоков кода» всё равно подавляет
  блоки кода. Если часть выхода должна быть исключена — перестрой так, чтобы правило до неё не дотянулось.

## RED-GREEN-REFACTOR для скиллов

| TDD | Создание скилла |
|-----|-----------------|
| Тест-кейс | Pressure-сценарий с субагентом |
| Production-код | SKILL.md |
| Тест падает (RED) | Агент нарушает правило БЕЗ скилла (baseline) |
| Тест проходит (GREEN) | Агент соблюдает при наличии скилла |
| Refactor | Закрыть лазейки, сохраняя compliance |

**Iron Law:** `NO SKILL WITHOUT A FAILING TEST FIRST`. Применяется и к НОВЫМ скиллам, и к ПРАВКАМ.
Написал скилл до теста — удали, начни заново.

**RED — Baseline:** прогнать pressure-сценарий БЕЗ скилла. Задокументировать дословно: какие решения,
какие рационализации, какое давление триггерит нарушение.
**GREEN — Минимальный скилл:** написать скилл, адресующий именно эти рационализации. Без контента «на всякий».
**REFACTOR — Закрыть лазейки:** новая рационализация → явный контр-аргумент → перетест до пуленепробиваемости.

### Micro-test перед полными сценариями
Полные pressure-прогоны — финальный гейт, но дороги. Сначала проверь саму формулировку:
1. Один свежий контекст на вызов (raw API-вызов или single-shot субагент). System = реалистичный контекст, где guidance будет жить; user = задача, соблазняющая на провал.
2. **Всегда включай no-guidance контроль.** Контроль не показывает провал → чинить нечего, стоп.
3. **5+ повторов на вариант.** Одиночные сэмплы врут.
4. **Читай глазами каждый flagged-матч** — эхо шаблона и цитаты контр-примеров маскируются под хиты.
5. **Variance — это метрика.** Пять разных интерпретаций на 5 повторов = формулировка не связывает, ужми форму.

## Bulletproofing против рационализации (для discipline-скиллов)

Только для discipline-провалов (агент знает правило и скипает под давлением). Для wrong-shape/omitted — см. таблицу выше.

1. **Закрой каждую лазейку явно.** Не просто «удали код до теста» → «Удали. Начни заново. Не держи как reference. Не адаптируй. Delete means delete.»
2. **Spirit vs Letter:** добавь рано `Нарушение буквы правила = нарушение духа правила.` — отрубает класс «я следую духу».
3. **Rationalization table:** каждую отмазку из baseline — в таблицу `| Отмазка | Реальность |`.
4. **Red Flags list:** сделай лёгким самопроверку («если думаешь X — STOP»).
5. **SDO под симптомы:** в `description` — симптомы момента, когда ты ВОТ-ВОТ нарушишь правило.

## SKILL.md: структура и discovery (SDO)

**Frontmatter:** `name` (буквы/цифры/дефисы), `description` — от третьего лица.

**⭐ description = КОГДА использовать, НЕ ЧТО делает.** Не пересказывай workflow скилла в description.
Тестирование показало: если description суммирует процесс, агент следует description ВМЕСТО чтения
тела скилла (описание «code review between tasks» заставило сделать ОДИН ревью вместо двух из flowchart).

```yaml
# ❌ Пересказ workflow — агент пойдёт по нему вместо чтения скилла
description: Use when executing plans - dispatches subagent per task with review between tasks
# ✅ Только триггеры
description: Use when executing implementation plans with independent tasks in the current session
```

- Ключевые слова, по которым будут искать (ошибки, симптомы, инструменты, синонимы).
- Активное имя, verb-first: `condition-based-waiting` > `async-test-helpers`.
- Token efficiency: часто-грузящиеся скиллы < 200 слов; детали — в отдельные файлы / `--help`.
- Cross-ref по имени с маркером: `**REQUIRED SUB-SKILL:** Use <skill>`. Никаких `@`-ссылок (force-load жжёт контекст).

**Flowchart — только** для неочевидных точек ветвления / циклов, где можно остановиться слишком рано.
НЕ для reference (таблицы), кода (markdown), линейных инструкций (списки).

## Наш reflection → rules контекст

Этот скилл — методология для ШАГ 5 (Post-Task) и Stage 6 (REFLECTION): когда урок превращается в
правило/скилл. Прежде чем дописать правило в `~/.claude/CLAUDE.md`, `WORKFLOW.md` или новый скилл:
1. Классифицируй baseline-провал (таблица Match the Form).
2. Выбери форму (prohibition / recipe / structural / conditional), НЕ по привычке.
3. Для discipline-правил — добавь rationalization-строку + red flag.
4. Идеально — micro-test формулировки на no-guidance контроле перед деплоем правила.

**Не деплой непротестированное правило так же, как не деплоишь непротестированный код.**

## Anti-patterns

- ❌ Narrative («в сессии 2026-01-10 мы нашли…») — не переиспользуемо; скилл ≠ история.
- ❌ Multi-language dilution (5 языков посредственно) — один отличный пример.
- ❌ Код внутри flowchart — нельзя скопировать.
- ❌ Generic labels (helper1, step3) — семантика обязательна.
- ❌ Description, пересказывающий workflow.
- ❌ Батч скиллов без теста каждого.

## Checklist (TDD-adapted)

**RED:** pressure-сценарии (3+ давления для discipline) → прогон БЕЗ скилла → задокументировать рационализации.
**GREEN:** `name` валиден · `description` = «Use when…» третьим лицом, без workflow · keywords · форма соответствует типу провала · behavior-shaping — micro-tested vs контроль · один отличный пример · прогон СО скиллом = compliance.
**REFACTOR:** новые рационализации → контры → rationalization table → red flags → перетест.
**Deploy:** коммит; если широко полезно — обновить индекс/CLAUDE.md.
